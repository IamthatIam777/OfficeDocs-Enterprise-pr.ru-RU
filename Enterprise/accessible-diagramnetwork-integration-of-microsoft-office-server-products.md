# <a name="accessible-diagram---network-integration-of-microsoft-office-server-products"></a>Доступная схема — сетевая интеграция серверных продуктов Microsoft Office

**Сводка:** Эта статья представляет собой текстовую версию схемы "Интеграция сети продуктов Microsoft Office Server".
  
На этом плакате представлена общая иллюстрация сетевой среды, включающей Lync Server 2013, SharePoint 2013 и Exchange Server 2013. В нем также показаны следующие элементы сети, общие для этих продуктов: удаленный и внутренний доступ, проверка подлинности, клиентский трафик и маршрутизация трафика через общие устройства. 
  
## <a name="high-level-concepts-for-lync-exchange-sharepoint-server-and-office-web-apps"></a>Высокоуровневые концепции для Lync, Exchange, SharePoint Server и Office Web Apps

### <a name="about-the-design"></a>Сведения о проектировании

#### <a name="streamlined-network-design"></a>Упрощенный дизайн сети

Эта топология иллюстрирует локальное развертывание сети для SharePoint 2013, Exchange Server 2013 и Lync Server 2013. Здесь также показано использование облачной службы Майкрософт, Exchange Online Protection, которая обеспечивает защиту от нежелательной почты и вредоносных программ для входящего трафика SMTP из Интернета. 
  
Эта структура сети оптимизирована для использования минимального набора сетевых компонентов. При проектировании не учитывается дополнительная безопасность или функции инфраструктуры, которые могут потребоваться некоторым организациям. 
  
Эта схема: 
  
- Пример сетевой топологии, иллюстрирующий входящий и исходящий трафик через маршрутизатор шлюза и балансировку нагрузки трафика клиентского сеанса (внешнего и внутреннего) на соответствующие уровни SharePoint, Exchange и Lync Server. 
    
- Показывает использование необязательных серверов удаленного доступа, таких как сторонний шлюз VPN или сервер DirectAccess, для обеспечения безопасной связи для перемещаемых или удаленных сотрудников. 
    
- Подробные сведения о потоке трафика SharePoint, Exchange и Lync от клиента к каждому уровню сервера платформы. 
    
- Определяет тип подключения удаленного или внутреннего доступа на основе клиента (например, "партнер" или "сотрудник") и используемого метода проверки подлинности. 
    
- Разбивает платформы SharePoint, Exchange и Lync с помощью необходимых ролей сервера, определяя интерфейсный сервер, приложение, базу данных и другие уровни. 
    
Архитектура, используемая в SharePoint, Lync и Exchange, не предлагает предпочтительный способ реализации этих платформ. Он просто предоставляет пример, так как топологии различаются на основе уникальных требований к сети и соображений безопасности. 
  
#### <a name="gateway-router"></a>Маршрутизатор шлюза

В этой топологии маршрутизатор шлюза располагается на границе сети и направляет весь входящий и исходящий трафик в интрасеть и из нее. Кроме того, могут быть также другие компоненты, которые приводят к мосту между маршрутизатором шлюза и подсистемой балансировки нагрузки, например с несколькими уровнями межсетевых экранов. Эта топология представляет только один способ развертывания сети из множества. В этой конфигурации шлюз настраивается с помощью списков управления доступом (ACL) для предоставления очень конкретного входящего и исходящего IP-трафика на интерфейсах маршрутизатора. Списки управления доступом, расширенная проверка и преобразование сетевых адресов (NAT) также могут быть выполнены на других устройствах, таких как брандмауэры, в сети. 
  
#### <a name="load-balancer-and-reverse-proxy-devices"></a>Подсистема балансировки нагрузки и устройства обратного прокси-сервера

Вы можете использовать аппаратные и программные решения балансировки нагрузки для перенаправления трафика для сегментов, включая интерфейсные веб-серверы SharePoint и серверы клиентского доступа Exchange (касс). В некоторых случаях рекомендуется использовать аппаратный балансировщик нагрузки для уровня 7, так как его можно улучшить с помощью сведений в запросе, например файлов cookie или заголовков. Тем не менее, такие факторы, как затраты и повышенное использование и Рабочая нагрузка из такого решения, могут быть нежелательны для конкретных потребностей. Рассмотрите следующие моменты для балансировки нагрузки в SharePoint, Exchange и Lync: 
  
- SharePoint для SharePoint 2013, нет необходимости включать сходство для интерфейсных веб-серверов. Обычно это используется для создания залипания сеансов и предотвращения нескольких запросов проверки подлинности от клиентов к каждому интерфейсному веб-серверу. Новая служба распределенного кэша в SharePoint 2013 хранит и распространяет маркеры входа на веб-серверах фермы SharePoint. 
    
- Exchange-in Exchange 2013, роль CAS разработана для использования балансировки нагрузки 4 уровня и распределения запросов на транспортном уровне. Это может значительно снизить загрузку и загрузку подсистемы балансировки нагрузки. 
    
- Lync — балансировка нагрузки системы доменных имен (DNS) рекомендуется для трафика SIP для пулов Lync. Аппаратная балансировка нагрузки (HLB) необходима для трафика Lync Web (HTTPS). 
    
### <a name="remote-access-options"></a>Варианты для удаленного доступа

Существует несколько параметров, которые могут публиковать ресурсы интрасети для партнеров в Интернете или обеспечивать безопасный удаленный доступ для удаленных или перемещаемых сотрудников. К таким примерам относятся обратные прокси-серверы, DirectAccess и сторонние VPN-шлюзы. Решения удаленного доступа, описываемые далее в этом разделе, это возможности SharePoint, Lync и Exchange, а также любая комбинация этих серверов в локальном развертывании. Однако некоторые удаленные параметры могут не работать с определенным решением. 
  
Обратный прокси — обратный прокси-сервер поддерживает шифрование трафика, например SSL, и с его помощью вы можете публиковать приложения интрасети и веб-ресурсы для пользователей и партнеров, прошедших проверку подлинности, в Интернете. В качестве примера используется Microsoft Forefront Unified Access Gateway (UAG). Многие аппаратные подсистемы балансировки нагрузки также поддерживают функции обратного прокси-сервера. Тем не менее существуют сценарии для использования автономного решения в зависимости от ваших потребностей и требований, таких как изоляция трафика, безопасность обособления и оптимизация производительности. 
  
Преимущества обратного прокси-сервера и рекомендации: 
  
- Предоставляет доступ к ресурсам интрасети для пользователей, прошедших проверку подлинности, и безопасный доступ к ресурсам интрасети (использует SSL (TCP 443) между клиентом и обратным прокси-сервером). 
    
- Для Exchange преимуществом использования обратного прокси-сервера, такого как Forefront UAG, является предварительная проверка подлинности перед доступом к серверу клиентского доступа Exchange. Пользователи удаленного доступа, использующие опубликованные приложения, такие как Outlook Web Access (OWA), могут выполнять проверку подлинности с помощью методов Basic, NTLM или Kerberos до того, как они достигают внутренней сети. 
    
- Для Exchange и SharePoint такие решения, как Forefront UAG, могут прерывать подключения SSL и снижать нагрузку на сервер ресурсов интрасети, в то же время обеспечивая единую точку управления для сертификатов. 
    
- Для сети Lync трафик через Интернет (HTTPS) проходит через обратный прокси-сервер (TCP 443) для связи с клиентом. Обратные прокси-серверы HTTPS-подключения к веб-службам Lync, Exchange CA и Office Web Apps. Lync Server 2013 не поддерживает UAG. 
    
DirectAccess — технология удаленного доступа, использующая безопасность протокола IP (IPsec) для проверки подлинности и шифрования трафика между клиентом и сервером DirectAccess. DirectAccess предоставляет одновременный доступ к ресурсам Интернета и интрасети для роуминга и удаленных сотрудников без необходимости запуска подключения. 
  
Факторы DirectAccess, которые следует рассмотреть: 
  
- DirectAccess использует защищенный трафик IPsec (протокол 50 и 51 и UDP 500) между клиентом и сервером DirectAccess. 
    
- Для DirectAccess для Windows Server 2012 и Windows 8 не требуется развертывание инфраструктуры открытых ключей (PKI) для проверки подлинности сервера и клиента. 
    
- Мы рекомендуем использовать DirectAccess с Lync Server 2013 из-за проблем с задержкой звука и видео, связанными с шифрованием и расшифрованием IPsec. 
    
    VPN-шлюз — типичные шлюзы VPN обеспечивают подключение удаленного доступа, в котором клиентский компьютер удаленного доступа логически проецируется на интрасеть через туннельное и инициированное пользователем подключение. Вы можете использовать единый удаленный доступ в Windows Server 2012 или нескольких сторонних решениях, чтобы обеспечить безопасный доступ к интрасети для перемещенных или удаленных сотрудников. VPN не рекомендуется для Lync. Удаленный трафик Lync должен использовать пограничные серверы и раздельное туннелирование. 
    
### <a name="domain-name-system-dns-considerations"></a>Вопросы, касающиеся системы доменных имен (DNS)

Необходимо запланировать набор записей DNS, позволяющих пользователям Интернета и интрасети разрешать DNS-имена в соответствующие IP-адреса. 
  
- Для партнеров в Интернете и перемещенных или удаленных сотрудников DNS-записи, зарегистрированные с помощью DNS-серверов в Интернете, предоставляют разрешение на набор общедоступных IP-адресов, соответствующих маршрутизатору шлюза, пограничному серверу Lync, набору виртуальных IP-адресов (VIP) на Подсистема балансировки нагрузки и шлюз DirectAccess или VPN по мере необходимости. 
    
- Для пользователей на основе интрасети записи DNS, зарегистрированные с помощью DNS-серверов интрасети, обеспечивают разрешение для набора виртуальных IP-адресов (VIP) в подсистеме балансировки нагрузки для доступа к ресурсам SharePoint, Lync и Exchange. 
    
- Клиенты DirectAccess используют DNS-серверы интрасети для имен, соответствующих пространству имен DNS в интрасети и DNS-серверам Интернета для имен, которые не поддерживаются. Чтобы упростить работу DirectAccess, рассмотрите возможность использования разделенной реализации DNS, использующей различные пространства имен DNS для имен интрасети и Интернета. Например, используйте contoso.com для пространства имен Интернета и corp.contoso.com для пространства имен интрасети. 
    
- Exchange использует разделенную модель DNS, в которой разрешение Host to IP отличается от трафика с общедоступным маршрутом из сети Организации. Как минимум, необходимо иметь записи DNS для URL-адресов OWA, автообнаружения, ActiveSync для клиентского трафика и запись MX для входящей почты. 
    
- Если вы используете Exchange Online Protection (EOP), запись MX указывает на эту службу, а не на ферму Exchange. 
    
- Для Exchange требуется проверка владельца записи TXT в общедоступной службе DNS, а также идентификатор Федерации для настройки федеративного общего доступа. 
    
- VPN-клиенты удаленного доступа могут быть настроены на использование только DNS-серверов в интрасети, когда VPN-подключение удаленного доступа активно.

#### <a name="description-of-how-traffic-moves-through-the-components-to-the-different-server-tiers"></a>Описание способа перемещения трафика через компоненты на различные уровни серверов

В этом разделе описывается, как запросы пользователей перемещаются по топологии сети. 
  
#### <a name="authentication-and-routing-for-external-access"></a>Проверка подлинности и маршрутизация для внешнего доступа

Существует три типа клиентов за прев своем сети и облачных службах, в том числе: 
  
- Партнерские компании (деловые отношения) — внешние 
    
- Отдельные партнеры (SharePoint и анонимные (Lync)) — внешние 
    
- Роуминг и удаленные сотрудники — внешние 
    
Процесс проверки подлинности и маршрутизации для каждого внешнего типа пользователей описывается по отдельности. 
  
#### <a name="partner-companies-business-to-business-httpspartnerwebcontosocom"></a>Партнерские компании (Business-to-предприятие) (https://partnerweb.contoso.com)

- Lync: доверие федерации с другими организациями, Skype и общедоступная служба обмена мгновенными сообщениями (PIC) с AOL. Трафик Федерации Lync проходит через маршрутизатор шлюза на пограничный сервер Lync, в виртуальный IP-адрес Lync (подсистема балансировки нагрузки/обратное прокси-сервер), а затем на сервер Lync Server. 
    
- SharePoint: доверенный поставщик удостоверений партнеров с проверкой подлинности SAML. Трафик клиента SharePoint проходит через маршрутизатор шлюза к виртуальному IP-адресу SharePoint (балансировщик нагрузки или обратный прокси-сервер), а затем к серверу SharePoint. 
    
- Exchange: взаимная проверка подлинности TLS для почтового трафика, проверка подлинности SAML для федеративного общего доступа. Трафик клиента Exchange проходит через маршрутизатор шлюза к виртуальному IP-адресу Exchange (балансировщик нагрузки/обратный прокси-сервер), а затем к серверу Exchange Server.

#### <a name="lync-sip-traffic-and-lync-web-traffic"></a>Трафик SIP Lync и веб-трафик Lync

Для взаимодействия с внешними пользователями пограничный сервер Lync использует следующие порты: 
  
- Передача сигналов/обмена мгновенными сообщениями (SIP/SIMPLE): TCP-порт 443 (открытая для входящего трафика) 
    
- Трафик веб-конференций (PSOM): TCP 443 (открыть для входящего трафика) 
    
- Поток аудио-и видеоданных (SRTP): TCP 443, UDP 3478 и TCP 50000-59999 (необязательно) (открытая для входящего и исходящего трафика). 
    
Для связи Федерации пограничный сервер Lync использует следующие порты: 
  
- TCP-порты 5061 (SIP), 5269 (XMPP), 443 и UDP 3478 (SRTP) (открытая для входящего и исходящего трафика). 
    
#### <a name="sharepoint-client-traffic"></a>Трафик клиента SharePoint

В SharePoint можно использовать TCP-порт 443 (SSL) для шифрования связи между клиентом и подсистемой балансировки нагрузки. Для внешнего доступа из Интернета этот порт должен быть открыт для входящего и исходящего трафика на маршрутизаторе шлюза (или внешнем брандмауэре). 
  
#### <a name="exchange-client-traffic-and-exchange-mail-server-traffic"></a>Трафик клиента Exchange и трафик почтового сервера Exchange

Exchange использует TCP-порт 25 (SMTP) для обмена данными между серверами. Большая часть клиентского трафика (OWA, ActiveSync, автообнаружение, мобильный Outlook) обрабатывается через порт 443 (HTTPS). Если у вас есть клиенты POP или IMAP, порты 110 (POP3), 995 (Encrypting POP3), 143 (IMAP4), 993 (encrypted IMAP4) и 587 (SMTP) также используются для поддержки этих клиентов.
