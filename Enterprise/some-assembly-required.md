# Решения, требующие сборки

 **Сводка.** Узнайте, как получить сведения о наборе параметров облачного хранилища, с помощью которых можно создать собственное решения для хранения.
  
Решения для хранения, требующие сборки:
  
- За основу берутся существующие службы.
    
- Требуется настройка или написание кода.
    
- Возможна настройка в соответствии с потребностями.
    
В следующих разделах подробно описывается каждое решения для хранения, требующее подготовки.

### Возможности

- База данных Stretch с SQL Server 2016
    
### Стандартные варианты использования

- Новые приложения с реляционными данными, разработанные для облака.
    
- Обработка данных в схематических высокоструктурированных наборах данных со связями.
    
- Данные сложных типов и пространственные данные.
    
### Основные сценарии использования хранилища

- Управление данными.
    
### Эластичная база данных

Используйте практически неограниченные ресурсы базы данных SQL Azure в следующих случаях:
  
- Общий объем данных слишком велик, чтобы разместить эти данные в одной базе данных.
    
- Количество транзакций в общей рабочей нагрузке превышает пропускную способность одной базы данных.
    
- Необходимо физически изолировать клиенты друг от друга, поэтому для каждого клиента нужна отдельная база данных.
    
- Для обеспечения соответствия требованиям, повышения производительности или из геополитических соображений необходимо размещать разные разделы базы данных в различных географических расположениях.
    
Вертикальное масштабирование: вы можете изменить уровень производительности или выпуск базы данных Azure либо использовать пулы эластичных баз данных.

Горизонтальное масштабирование: при необходимости вы можете добавлять новые базы данных.

### База данных Stretch с SQL Server 2016

База данных Stretch  это функция SQL Server 2016, которая позволяет прозрачно и безопасно переместить редко используемые данные, например данные о закрытой компании в большой таблице со сведениями о заказах клиентов, в базу данных Stretch в SQL Azure. При использовании функции Stretch содержимое экземпляра SQL Server, база данных или даже одна таблица являются сочетанием локальных данных на сервере SQL Server 2016 и удаленных данных в Azure. SQL Server 2016 автоматически перемещает в Azure данные, которые поддерживают функцию Stretch.
  
![База данных Stretch с SQL Server 2016.](images/b5360311-8100-42e9-ac62-9b611294a34d.png)
  
 Запросы пользователей, которые содержат архивные данные, прозрачно перенаправляются в базу данных Stretch в SQL Azure. Несмотря на то что таблица растягивается, запросы не нужно записывать повторно.
  
База данных Stretch представляет собой экономически эффективное решение для долгосрочного хранения архивных данных и прозрачного доступа к ним. Кроме того, она позволяет решить проблемы производительности и доступности, возникающие при значительном увеличении таблиц.
  
Дополнительные сведения вы найдете [здесь](https://msdn.microsoft.com/library/dn935011.aspx).

## Кэш Redis для Azure

### Возможности

- Защищенный выделенный сервер Redis с функциями обеспечения высокой доступности, репликации данных и отработки отказа, управляемый корпорацией Майкрософт.
    
- Рекомендуется для любых приложений, которым необходима высокая пропускная способность.
    
- Доступны размеры до 530 ГБ и более (для уровня Premium и автоматической сегментации).
    
- Функция Redis Persistence сохраняет кэшированные в памяти данные в службу хранилища Azure.
    
- Функция Redis Clustering позволяет достичь максимальных масштаба и пропускной способности.
    
- Усиленная безопасность и изоляция сети с поддержкой виртуальной сети Azure.
    
### Стандартные варианты использования

- Обратный поиск данных в любой службе хранения в Azure, например в Cosmos DB и базе данных SQL Azure.
    
- Синхронизированное содержимое из других хранилищ данных.
    
### Основные сценарии хранения

- Кэширование данных.
    
- Брокер сообщений для приложений, требующих высокой пропускной способности.
