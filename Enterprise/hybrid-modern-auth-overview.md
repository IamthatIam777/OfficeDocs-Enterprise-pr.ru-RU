## <a name="what-is-modern-authentication"></a>Что такое современных проверка подлинности?
<a name="BKMK_WhatisModAuth"> </a>

Когда речь идет о связи между клиентом (например, ноутбука или телефон) и сервером, корпорация Майкрософт использует фразу «Современный проверки подлинности».
  
Современный проверки подлинности — это общий термин для сочетание проверки подлинности и авторизации методы, а также некоторые меры безопасности, зависящие от политики доступа, которые вы уже быть знакомы с. Оно включает в себя:
  
- **Методы проверки подлинности**: многофакторная проверка подлинности; Проверка подлинности на основе сертификата клиента; и библиотеки проверки подлинности Active Directory ( [ADAL](https://technet.microsoft.com/en-us/library/mt710548.aspx)).
    
- **Методы авторизации**: реализация корпорации Майкрософт Open Authorization (OAuth). 
    
- **Политики условного доступа**: мобильные приложения управления (MAM) и Azure Active Directory условного доступа. 
    
Управление удостоверениями пользователей с проверкой подлинности на современном предоставляет в распоряжение администраторов много различных инструментов для использования при его в вопросах безопасности ресурсов и более безопасной методы управления удостоверениями, чтобы в локальной (Exchange и Скайп для бизнеса), Exchange гибридного и Скайп для бизнес-сценарии гибридных/split домена.
  
Обратите внимание, что так как Скайп для бизнеса работает в тесном сотрудничестве с сервером Exchange, для входа поведение Скайп для клиента Business пользователи увидят будет повлиять состояние современных проверки подлинности Exchange. Это также будут применяться, если у вас есть Скайп для гибридного домена разделения бизнеса. Кроме того, тип Скайп для бизнеса гибридной среды, который поддерживает использование проверки подлинности на современном часто называется «split домен» (split-домена, у вас есть Скайп для бизнеса в Интернет и Скайп для бизнеса на prem и пользователей, размещенных в обоих местах).
  
 **Важные** Вам известно, что по состоянию на август 2017 всех новых клиентов Office 365, которые включают Скайп для бизнеса через Интернет и Exchange online будут иметь современных проверки подлинности по умолчанию? Существующие клиенты не был изменен в состоянии агент Управления по умолчанию, но всех новых клиентов автоматически поддерживает расширенный набор идентификаторов функций перечисленных выше. Для проверки состояния агента Управления Скайп для бизнеса через Интернет, можно использовать Скайп для бизнеса online PowerShell с использованием учетных данных глобального администратора. Запустите «Get-CsOAuthConfiguration» проверка вывода - ClientADALAuthOverride. Если - ClientADALAuthOverride «разрешен» современный подлинности включен. 
  
## <a name="what-changes-when-i-use-modern-authentication"></a>Какие изменения происходят при использовании проверки подлинности на современном?
<a name="BKMK_WhatChanges"> </a>

При использовании проверки подлинности современных с локальной Скайп для бизнеса или Exchange server, вы по-прежнему *проверки подлинности* пользователей в локальной, но История *авторизации* их доступа к ресурсам (например, файлы или по электронной почте) изменения. Именно поэтому на то, что современных проверки подлинности клиента и сервера соединения, действия, предпринятые во время настройки результатов агент Управления в evoSTS (служба маркеров безопасности используется Azure AD) установить в качестве сервер проверки подлинности для Скайп для бизнеса и Exchange server в локальной. 
  
Изменение evoSTS позволяет локальную серверов, чтобы воспользоваться преимуществами OAuth (выдача маркеров) для авторизации вашим клиентам и также позволяет локального использования безопасности методов распространенных в облаке (например, многофакторная проверка подлинности). Кроме того evoSTS маркеры, которые позволяют пользователям запрашивать доступ к ресурсам без указания свои пароли в запросе. Независимо от того, где размещаются пользователи (версией через Интернет или локальной), и независимо от того, какие местоположения размещается необходимых ресурсов, EvoSTS станут основных авторизации пользователей и клиентов, после настройки современных проверки подлинности.
  
Ниже приведен пример я понимаю. Если Скайп для клиента Business требуется доступ к Exchange server для получения данных календаря от имени пользователя, для этого используется библиотеки проверки подлинности Active Directory (ADAL). ADAL библиотеку кода позволяет сделать доступным для клиентских приложений с помощью маркеров безопасности OAuth защищенным ресурсам в каталоге. ADAL работает с OAuth для проверки утверждений и обмениваться маркерами (а не паролей), чтобы предоставить пользователю доступ к ресурсам. В прошлом центра сертификации в транзакции аналогичных этому--сервера, который знает, как проверить пользователя утверждений и выдача необходимые маркеры — был службы маркеров безопасности в локальной или даже служб федерации Active Directory. Тем не менее современный проверки подлинности позволяет централизовать эту службу с помощью Azure Active Directory (Azure AD) в облаке.
  
Это также означает, что несмотря на то что Exchange server и Скайп для бизнес-сред может быть полностью локальной, авторизацию сервер будет online и локальной среде необходимо иметь возможность создавать и настраивать подключения к Office 365 подписки в облаке (и экземпляр Azure Active Directory, использующий подписки в качестве его каталог).
  
Что не изменяется при? Ли вы гибридного домена разделения или с помощью Скайп для бизнеса и Exchange server в локальной, все пользователи сначала должно проходить проверку подлинности *в локальной* . В реализации гибридных современных проверки подлинности Lyncdiscovery и автоматического обнаружения наведите указатель на локальном сервере. 
  
 **Важные** Если необходимо знать определенные Скайп для Business топологии, поддерживаемые с помощью агента Управления, который является [описанной здесь справа](https://technet.microsoft.com/en-us/library/mt803262.aspx).
  
## <a name="check-the-modern-authentication-status-of-your-on-premises-environment"></a>Проверка состояния современных проверки подлинности в локальной среде
<a name="BKMK_CheckStatus"> </a>

Поскольку современных проверки подлинности изменяется сервера авторизации, который используется при служб, использующих возможности OAuth/S2S, необходимо знать, если современных проверки подлинности будет включено или отключено для вашей Скайп для среды Exchange и бизнес. Можно проверить состояние на Exchange или Скайп на серверах предприятия, при локальном, с помощью команды Get-CSOAuthConfiguration в PowerShell. Если команда возвращает пустой свойство «OAuthServers», современный проверка подлинности отключена.
