# <a name="accessible-diagram---internet-sites-in-microsoft-azure-for-sharepoint-2013"></a>Доступная схема — веб-сайты в Microsoft Azure для SharePoint 2013

**Сводка:** Эта статья является доступным текстовая версия схемы с именем веб-сайтов в Microsoft Azure для SharePoint 2013.
  
На этом плакате рассказано и показано, какие преимущества для общедоступных веб-сайтов дают гибкие облачные решения и использование Azure Active Directory для учетных записей клиентов. На плакате имеется шесть указанных ниже сценариев, d которых рассказывается, какие преимущества получают веб-сайты от использования Azure.  
  
- Разработка и выбор размера топологии фермы.  
    
- Настройка для Azure.  
    
- Выбор модели Active Directory.  
    
- Планирование функций управления удостоверениями, работы с зонами и проверки подлинности.  
    
- Планирование сайтов и URL-адресов для межсайтовой публикации.  
    
- Разработка среды Azure.  
    
## <a name="design-and-size-the-farm-topology"></a>Разработка и выбор размера топологии фермы

Используйте рекомендации топологии, емкости и производительности для SharePoint 2013 на сайте TechNet для построения топологии фермы. 
  
Необходимо, чтобы разрабатываемая вами ферма соответствовала требованиям к мощности и производительности.  
  
### <a name="example-medium-internet-sites-farm-85-page-views-per-second"></a>Пример: ферма среднего размера для веб-сайтов (около 85 просмотров страниц в секунду)

Эта ферма предоставляет отказоустойчивой топологией фермы поиска SharePoint 2013, который оптимизирован для собрания документов, содержащей 3,400,000 элементы. 
  
Пример фермы обрабатывает 100-200 документов в секунду, в зависимости от языка, и обеспечивает 85 просмотров страниц в секунду и 100 запросов в секунду. 
  
На сопроводительной схеме показана ферма среднего размера для веб-сайтов с тремя указанными ниже типами серверов.  
  
- Веб-серверы 
    
- Серверы приложений 
    
- Серверы баз данных 
    
#### <a name="web-servers"></a>Веб-серверы

В разделе веб-серверов имеются три веб-сервера: "Узел A", "Узел B" и "Узел C". На каждом веб-сервере есть распределенный кэш, профили пользователей, управляемые метаданные и средства обработки запросов. Кроме того, они содержат реплику раздела индекса, имеющегося в каждом сервере.   
  
Для горизонтального масштабирования службы добавьте дополнительный веб-сервер с такими же возможностями, позволяющий увеличить скорость обработки на 28 просмотров страниц в секунду.  
  
#### <a name="application-servers"></a>Серверы приложений

В разделе серверов приложений есть три сервера приложений: "Узел D", "Узел E" и "Узел F". "Узел D" содержит компоненты обхода контента, администрирования, аналитики и обработки контента. "Узел E" содержит компоненты обхода контента, администрирования и обработки контента. "Узел F" содержит компоненты обхода и обработки контента.  
  
Для горизонтального масштабирования службы добавьте один сервер приложений с компонентами обхода и обработки контента, чтобы увеличить скорость обработки на 40 документов в секунду.  
  
#### <a name="database-servers"></a>Серверы баз данных

В разделе серверов баз данных имеются два сервера: "Узел G" и "Узел H". Для обеспечения отказоустойчивости серверы баз данных располагаются в сопряженных узлах.  
  
"Узел G" содержит все базы данных SharePoint, включая базу данных администрирования поиска, базу данных ссылок, две базы данных обхода контента, базу данных аналитики и все остальные базы данных SharePoint. "Узел H" содержит все базы данных SharePoint, включая избыточные копии всех баз данных, использующих кластеризацию и зеркальное отображение SQL или SQL Server 2012 AlwaysOn.  
  
## <a name="fine-tune-for-azure"></a>Настройка для Azure

Возможно, вам потребуется настроить ферму SharePoint для групп доступности на платформе Azure. Чтобы обеспечить высокую доступность всех компонентов, убедитесь, что все роли серверов настроены одинаково. 
  
В примере топологии на схеме:  
  
- веб-серверы и серверы баз данных настроены одинаково;  
    
- три сервера приложений настроены по-разному. Необходимо настроить эти роли серверов для групп доступности в Azure. 
    
### <a name="before"></a>До

В верхней части схемы показана ферма SharePoint до ее настройки для групп доступности в Azure. На схеме три сервера приложений настроены по-разному. Количество компонентов определяется требуемыми производительностью и мощностью фермы. Эти три сервера настроены указанным ниже образом.  
  
- Для сервера приложений "Узел D" настроены роли обхода контента, администрирования, аналитики и обработки контента.  
    
- Для сервера приложений "Узел E" настроены роли обхода контента, администрирования и обработки контента.  
    
- Для сервера приложений "Узел F" настроены роли обхода и обработки контента.  
    
### <a name="after"></a>После

Это часть схемы отображает в ферме SharePoint, после его были настройки доступности задается в Azure. Эта архитектура адаптировать для Azure, мы реплицировать четыре компонента для всех трех серверов. Это увеличивает число компонентов возможности, необходимые для повышения производительности и емкости. Компромисс — это гарантирует, что этот конструктор высокой доступности всех четырех компонентов в платформе Azure при назначении эти три виртуальных машин в набор доступности. 
  
Все три сервера имеют роли обхода контента, администрирования, аналитики и обработки контента.  
  
## <a name="choose-the-active-directory-model"></a>Выбор модели Active Directory

Для всех решений SharePoint требуются доменные службы Windows Active Directory. В настоящий момент существует два варианта решений SharePoint в Azure.   
  
- Вариант 1: Выделенного домена — выделенные и изолированный домен можно развернуть в Azure для поддержки фермы SharePoint. Это хороший выбор общедоступный веб-сайтов. 
    
- Вариант 2: Расширение локального домена через VPN-подключение веб сайта. При расширении локального домена через VPN-подключение веб сайтов, доступ к ферме SharePoint как для размещенных в локальной. Вы можете воспользоваться преимуществами существующей реализации Active Directory и DNS. 
    
## <a name="design-for-identity-management-zones-and-authentication"></a>Планирование функций управления удостоверениями, работы с зонами и проверки подлинности

### <a name="design-for-accounts-and-authentication"></a>Учетные записи и проверка подлинности

Определите, как будет выполняться управление учетными записями и какой тип проверки подлинности будет использоваться.  
  
#### <a name="accounts-for-site-developers-and-authors"></a>Учетные записи для разработчиков сайта и авторов

- Добавьте учетные записи к домену в Azure.  
    
- Используйте локальные службы федерации Active Directory для создания федерации внутренних учетных записей с доменом в Azure.  
    
- Если схема включает VPN-подключение типа "сеть-сеть", используйте внутренние учетные записи.  
    
#### <a name="accounts-for-customers"></a>Учетные записи для клиентов

- Используйте Azure Active Directory. 
    
- Используйте другого поставщика на основе SAML.  
    
### <a name="design-for-zones"></a>Зоны

В SharePoint 2013 управление удостоверениями учитывается в конфигурации зон и при проверке подлинности.  
  
Такая схема обеспечивает четкое отделение учетных записей клиентов от всех остальных учетных записей.  
  
- Используйте эту схему, если необходимо обрабатывать учетные записи клиентов совершенно другим способом, чем внутренние учетные записи для авторов и разработчиков сайта.  
    
- Эта схема позволяет использовать политики зон для ограничения действий клиентов в веб- приложении.  
    
- При использовании этой схемы учетные записи клиентов и внутренние учетные записи будут иметь разные URL-адреса.  
    
В этом примере: 
  
- настройка зоны по умолчанию для внутренних учетных записей; 
    
- настройка зоны экстрасети для доступа клиентов с проверкой подлинности. Для учетных записей клиентов используйте службу Microsoft Azure Active Directory или другого поставщика на основе SAML;  
    
- настройка зоны Интернета для анонимного доступа. 
    
Не используйте разработки двумя зонами, в которой все пользователи, прошедшие проверку настроены на использование зоны по умолчанию. 
  
На сопроводительной схеме показана схема с тремя зонами и отделением внутренних учетных записей от учетных записей клиентов.   
  
Посетители и клиентам доступа клиента AD Azure в ферме SharePoint 2013 через веб-приложений в одном из двух зон. Двух зон, относятся: 
  
- Зона: Интернет для анонимных пользователей  
    
- Зона: экстрасеть для пользователей, прошедших проверку подлинности  
    
Пользователи с внутренними учетными записями получают доступ к клиенту Azure Active Directory из доменных служб Active Directory и служб федерации Active Directory в локальной среде через VPN-туннель к Azure Active Directory. Зона по умолчанию обеспечивает проверку подлинности Windows (NTLM).  
  
### <a name="design-for-connecting-to-azure-ad"></a>Подключение к Azure Active Directory

 Служба Azure Active Directory предоставляет возможности управления удостоверениями и доступом для облачных служб. Вы сможете использовать облачное хранилище для данных каталогов и базовый набор служб идентификации, в том числе процессы входа пользователей и службы проверки подлинности, а также службы федерации Active Directory. Службы идентификации, входящие в состав Azure Active Directory, можно легко интегрировать с локальными развертываниями доменных служб Active Directory. Они обеспечивают полную поддержку сторонних поставщиков удостоверений.
  
На сопроводительной схеме показан указанный ниже сценарий.  
  
При установке SharePoint 2013 с помощью Azure Active Directory, Azure Access Control Service (ACS) выполняет две функции: 
  
-   Azure Active Directory использует SAML 2.0, а SharePoint поддерживает только SAML 1.1. Служба контроля доступа поддерживает оба формата и выступает в роли посредника, преобразовывая форматы маркеров, передаваемых между SharePoint и Azure AD.   
    
- Служба контроля доступа позволяет не использовать службу маркеров безопасности поставщика удостоверений (IP-STS) для этого сценария SAML.  
    
Дополнительные сведения см. в статье "Настройка Azure Active Directory с SharePoint 2013" в библиотеке TechNet.  
  
## <a name="design-sites-and-urls-for-cross-site-publishing"></a>Планирование сайтов и URL-адресов для межсайтовой публикации

Схема с одним веб-приложением рекомендуется для сценариев публикации.  
  
- Сайты разработки и публикации находятся в одном и том же веб-приложении.  
    
- Для публикации активов используется функция межсайтовой публикации.  
    
Использование семейств веб-сайтов с именами на основе путей и узлов. 
  
- Необходимо корневое семейство веб-сайтов. Создайте его в виде сайта с именем на основе пути.  
    
- Создайте все остальные семейства веб-сайтов с именами на основе узла.  
    
URL-адреса веб-приложения и корневого сайта  
  
- В качестве URL-адреса веб-приложения используйте внутреннее имя. В качестве имени по умолчанию SharePoint использует имя локальной машины, если не указано другое имя. Вы можете использовать доменное имя, зарезервированное для внутренней сетевой среды.   
    
- При создании веб-приложения SharePoint назначает нестандартный номер порта. Используйте этот номер порта вместо порта 80 или 443. Кроме того, можно использовать другой нестандартный номер порта.  
    
- Используйте те же имя и номер порта для корневого семейства веб-сайтов с именем на основе пути.  
    
На сопроводительной схеме показаны службы пула приложений (например, служба поиска), взаимодействующие с семействами веб-сайтов с помощью веб-приложений. Ниже перечислены показанные на схеме семейства веб-сайтов.  
  
- Семейство веб-сайтов с именем на основе пути, расположенное по адресу http://internal:8000 (корневой сайт).  
    
- Обход контента Семейство веб-сайтов с именем на основе узла, расположенное, например, по адресу https://authoring.contoso.com:8000.  
    
- Запросы Два отдельных семейства веб-сайтов с именами на основе узлов, расположенные, например, по указанным ниже адресам.  
    
  - http://www.contoso.com 
    
  - https://Secure.contoso.com 
    
  - http://www.contoso.com:8000 
    
  - http://Assets.contoso.com 
    
  - https://secureassets.contoso.com 
    
  - http://Assets.contoso.com:8000 
    
## <a name="design-the-azure-environment"></a>Разработка среды Azure

Этот пример архитектуры включает указанные ниже элементы.  
  
- Необязательное VPN-подключение типа "сеть-сеть" позволяет расширить локальную среду доменных служб Active Directory и DNS, связав ее с виртуальной сетью в Azure.  
    
- При желании для поддержки фермы SharePoint можно использовать выделенный домен в Azure.  
    
- Серверы распределены между облачными службами Azure в зависимости от их ролей.  
    
- Группы доступности обеспечивают высокую доступность серверов, для которых настроены одинаковые роли.   
    
Дополнительные сведения см. в следующей статье на веб-сайте TechNet: "Архитектуры Azure для решений SharePoint".  
  
На сопроводительной схеме показан пример локальной среды, подключенной к виртуальной сети Azure.    
  
В локальную среду, являющуюся необязательным элементом среды Azure, включены указанные ниже компоненты.  
  
- Windows Server 2012 RRAS 
    
- Доменные службы Active Directory 
    
- VPN-шлюз, отделяющий Windows Server и доменных служб Active Directory от подсети активного VPN-шлюза  
    
Среда виртуальной сети Azure включает указанные ниже компоненты.  
  
- Подсеть активного VPN-шлюза (при необходимости перекрывающаяся с локальной средой)  
    
- Облачная служба, включающая доменные службы Active Directory и группу доступности DNS (два сервера)  
    
- Облачная служба, включающая группу доступности сервера переднего плана (три сервера SharePoint) и группу доступности сервера приложений App (три сервера SharePoint)  
    
- Облачная служба, содержащая две группы доступности баз данных
