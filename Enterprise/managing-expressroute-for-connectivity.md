# <a name="managing-expressroute-for-office-365-connectivity"></a>Управление подключением ExpressRoute для Office 365

ExpressRoute для Office 365 предлагает альтернативный путь маршрута для достижения многим службам Office 365 без применения какого-трафик для исходящего трафика в Интернет. Несмотря на то, что по-прежнему требуется подключение к Интернету в Office 365, определенные маршруты, которые Microsoft объявление посредством протокола BGP для вашей сети сделать прямого канала ExpressRoute предпочитаемое только при наличии других конфигураций в вашей сети. Три общие области, которые можно настроить для управления этой маршрутизации являются префикс фильтрации, безопасности и соответствия требованиям.
  
> [!NOTE]
> Microsoft изменены как оценку домена маршрутизации Microsoft Peering для Azure ExpressRoute. Запуск 31 июля 2017, все клиенты Azure ExpressRoute можно включить Microsoft Peering, непосредственно из консоли администрирования Azure или через PowerShell. После включения Microsoft Peering любого клиента, можно создать фильтры маршрутов для получения объявления маршрутов протокола BGP для сотрудничества с клиентами Dynamics 365 приложений (ранее называлась CRM Online). Клиенты, которым требуется Azure ExpressRoute для Office 365 необходимо получить review корпорацией Майкрософт перед могут создавать фильтры маршрутов для Office 365. Обратитесь в группу учетную запись Майкрософт для получения сведений о том, как запросить Проверка включения Office 365 ExpressRoute. Несанкционированная подписок для создания фильтров маршрутов для Office 365 получите [сообщение об ошибке](https://support.microsoft.com/kb/3181709)
  
## <a name="prefix-filtering"></a>Префикс фильтрации

Корпорация Майкрософт рекомендует пользователям принимать все маршруты протокола BGP как объявленных корпорацией Майкрософт, предоставленных маршруты подвергаются строгие процесс проверки и проверки, удаление каких-либо преимуществ для добавлена к критическому анализу. Рекомендуемые элементы управления, таких как IP-адрес префикс владельца, целостности и масштаба - изначально предлагает ExpressRoute с входящей маршрутизации не фильтрации на стороне клиента.
  
Если вам требуется дополнительная проверка владения маршрута между ExpressRoute общедоступных авторами, вы можете проверить заявленным маршрутов со списком всех IPv4 и IPv6-префиксы, представляющих [корпорации Майкрософт общедоступных диапазонов IP-адресов](https://www.microsoft.com/download/details.aspx?id=53602). Эти диапазоны охватывают полный адресного пространства Microsoft и измените редко, предоставляя надежный набор диапазонов для фильтрации по, также обеспечивает дополнительную защиту для клиентов, которые беспокоятся о сторонних владении маршрутов утечки в их Среда. В случае изменяется, будут создаваться на 1st месяца и номер версии в разделе **сведения о** странице изменится каждый раз при обновлении файла.
  
Существует ряд причин, чтобы не использовать [IP-адресов и URL-адреса Office 365 диапазонов адресов](https://aka.ms/o365endpoints) для создания списков фильтров префикса. Включая:
  
- Префиксы IP-адресов для Office 365 подвергаются большое число изменений частое.

- URL-адреса Office 365 и IP-адрес диапазонов предназначены для управления брандмауэра разрешить списки и инфраструктуры прокси-сервер не маршрутизации.

- Адрес диапазоны IP-адресов и URL-адреса Office 365 не охватывают другие службы Майкрософт, которые могут находиться в области ExpressRoute подключений.

| |
|**Вариант**|**Сложность**|**Управление изменениями**|
|:-----|:-----|:-----|
|Принимать все маршруты Microsoft  <br/> |**Low:** Клиент полагается на элементы управления Microsoft, чтобы убедиться, что все маршруты принадлежат должным образом.  <br/> |Нет  <br/> |
|Filter Microsoft владении supernets  <br/> |**Среднего:** Клиент реализует списки фильтров обобщенное префикс разрешить только Microsoft, владельцем которого маршрутов.  <br/> |Пользователям необходимо убедиться, что относительно редкие процедуры объединения обновлений вступили в силу в фильтрах маршрут.  <br/> |
|Диапазоны IP-адресов для Office 365 фильтров  <br/> [!CAUTION] Не рекомендуется
|**Высокой:** Клиент фильтр маршрутов на основании заданных префиксы IP-адресов для Office 365.  <br/> |Пользователям необходимо реализовать процесс управления изменениями надежную ежемесячный обновлений.  <br/> [!CAUTION]В этом решении требует значительных изменений постоянным. Изменения не реализованы в времени могут измениться недоступности службы.   |

Подключение к Office 365 с помощью Azure ExpressRoute выполняется на основе протокола BGP рекламы конкретных IP-подсети, которые представляют сетей, где развернут конечных точках Office 365. Из-за универсальность Office 365 и число служб, которые будут включены в Office 365 клиенты могут часто потребность в управлении объявления, которое они принять во внутренней сети. Если вы хотите с номером префиксов, объявленных в вашей среде, функция [протокола BGP сообщества](https://support.office.com/article/Using-BGP-communities-in-ExpressRoute-for-Office-365-scenarios-preview-9ac4d7d4-d9f8-40a8-8c78-2a6d7fe96099) позволяет фильтровать рекламные к определенному набору служб Office 365. Эта функция является теперь в режиме предварительного просмотра.
  
Независимо от того, как управлять объявления маршрутов протокола BGP, поступающих от корпорации Майкрософт не будут получать все специальные доступом к службам Office 365 по сравнению с подключение к Office 365 через Интернет каналы сам по себе он. Корпорация Майкрософт поддерживает же безопасности, соответствия требованиям и уровни производительности независимо от типа канала, который клиент использует для подключения к Office 365.
  
### <a name="security"></a>Безопасность

Корпорация Майкрософт рекомендует поддерживать собственные сети и безопасности периметра элементы управления для подключения к более подробное между ExpressRoute общедоступных и авторами Microsoft включающая подключений в службах Office 365. Элементы управления безопасности должны соблюдаться сетевых запросов, которые ездить исходящих из локальной сети, в сети корпорации Майкрософт, а также входящий трафик из сети корпорации Майкрософт в вашей сети.
  
#### <a name="outbound-from-customer-to-microsoft"></a>Исходящий трафик из клиента в корпорацию Майкрософт
  
Когда компьютеры подключение к Office 365, они подключаются к один и тот же набор конечных точек, независимо от того, ли выполнено подключение через Интернет или канал ExpressRoute. Независимо от того, канал используется Корпорация Майкрософт рекомендует обрабатывать служб Office 365, как надежные более чем универсальный назначения Интернета. Безопасность исходящих подключений элементы управления должны быть направлены на порты и протоколы для снижения риска и свести к минимуму текущего обслуживания. Сведения о необходимых портов доступна в справочная статья [конечных точках Office 365](https://aka.ms/o365endpoints) .
  
Для добавленных элементов управления можно использовать фильтрацию в инфраструктуре прокси-сервер уровня полного доменного ИМЕНИ ли ограничить или изучение некоторых или всех сетевых запросов, предназначенных для Интернета или Office 365. Обслуживание список полных доменных имен, как будут отпущены функции и развиваться услуги Office 365 требуется более надежная управления изменениями и отслеживание изменений опубликованного [конечных точках Office 365](https://aka.ms/o365endpoints).
  
> [!CAUTION]
> Корпорация Майкрософт рекомендует не полагаться на префиксы IP-адресов для управления безопасность исходящей почты в Office 365.

|**Вариант**|**Сложность**|**Управление изменениями**|
|:-----|:-----|:-----|
|Без ограничений  <br/> |**Low:** Клиент позволяет неограниченный доступ исходящих в корпорацию Майкрософт.  <br/> |Нет  <br/> |
|Ограничения портов  <br/> |**Low:** Клиента ограничивает исходящий доступ в корпорацию Майкрософт, ожидаемый порты.  <br/> |Относительно редкие процедуры объединения.  <br/> |
|Полное доменное имя ограничения  <br/> |**Высокой:** Клиент ограничивает исходящий доступ к Office 365 на основании опубликованные полных доменных имен.  <br/> |Месячная изменения.  <br/> |

#### <a name="inbound-from-microsoft-to-customer"></a>Входящий трафик от корпорации Майкрософт для клиента
  
Существует несколько сценариев необязательно, которым требуется Microsoft для инициации подключения по локальной сети.
  
- Службы федерации Active Directory во время подтверждения пароля для входа в систему.

- [Exchange Server гибридные развертывания](https://technet.microsoft.com/library/jj200581%28v=exchg.150%29.aspx).

- Почта от Exchange Online клиента для локального узла.

- SharePoint Online почты отправить из SharePoint Online локального узла.

- [Федеративные гибридного поиска SharePoint](https://technet.microsoft.com/library/dn197174.aspx).

- [Гибридная среда SharePoint BCS](https://technet.microsoft.com/library/dn197239.aspx ).

- [Скайп для гибридных бизнеса](https://technet.microsoft.com/en-us/library/jj205403.aspx) и/или [Скайп для федерации бизнеса](https://technet.microsoft.com/library/skype-for-business-online-federation-and-public-im-conectivity.aspx).

- [Скайп для Cloud Business Connector](https://technet.microsoft.com/library/mt605227.aspx ).

Корпорация Майкрософт рекомендует принимать эти подключения через вашей каналы Интернета вместо вашего ExpressRoute канала для уменьшения сложности. Если соответствие требованиям или производительности должен определения того, что эти входящих подключений необходимо принять через канал ExpressRoute, рекомендуется использовать брандмауэр или обратного прокси-сервера для определения области принятых подключений. Чтобы выяснить, право полные доменные имена и IP-адрес префиксы можно использовать [конечных точках Office 365](https://aka.ms/o365endpoints) .
