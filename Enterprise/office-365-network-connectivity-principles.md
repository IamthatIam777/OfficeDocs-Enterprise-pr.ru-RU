# <a name="microsoft-365-network-connectivity-principles"></a>Принципы сетевого подключения к Microsoft 365

*Эта статья относится как к Microsoft 365 Enterprise, так и к Office 365 корпоративный.*

Прежде чем приступить к планированию сети для Microsoft 365, важно понимать принципы подключения для безопасного управления трафиком Microsoft 365 и получения максимальной производительности. Эта статья поможет вам ознакомиться с последними рекомендациями по безопасной оптимизации сетевого подключения Microsoft 365.
  
Традиционные корпоративные сети предназначены в первую очередь для предоставления пользователям доступа к приложениям и данным, размещенным в обслуживаемых компанией центрах обработки данных с усиленной безопасностью периметра. Традиционная модель предполагает, что пользователи будут получать доступ к приложениям и данным из периметра корпоративной сети, через WAN-каналы филиалов или удаленно через VPN-подключения.
  
Внедрение приложений SaaS, таких как Microsoft 365, перемещает некоторую комбинацию служб и данных вне периметра сети. Без оптимизации трафик между пользователями и приложениями SaaS подвергается задержке, представленному в ходе проверки пакетов, сети развороты, непреднамеренного подключения к географически удаленным конечным точкам и другим факторам. Вы можете обеспечить оптимальную производительность и надежность Microsoft 365, поняв и реализуя ключевые рекомендации по оптимизации.
  
В этой статье вы узнаете о следующих возможностях:
  
- [Архитектура Microsoft 365](office-365-network-connectivity-principles.md#BKMK_Architecture) , которая применяется к подключению клиентов к облаку
- Обновлены принципы и стратегии [подключения Microsoft 365](office-365-network-connectivity-principles.md#BKMK_Principles) для оптимизации сетевого трафика и взаимодействия с конечными пользователями
- [Веб-служба конечных точек Office 365](office-365-network-connectivity-principles.md#BKMK_WebSvc), которая позволяет сетевым администраторам использовать структурированный список конечных точек для использования в оптимизации сети
- Новые категории и рекомендации по оптимизации для [конечных точек Office 365](office-365-network-connectivity-principles.md#BKMK_Categories)
- [Сравнение безопасности периметра сети с безопасностью конечной точки](office-365-network-connectivity-principles.md#BKMK_SecurityComparison)
- Параметры [добавочной оптимизации](office-365-network-connectivity-principles.md#BKMK_IncOpt) для трафика Microsoft 365
- [Проверка подключения microsoft 365](https://aka.ms/netonboard)— новое средство для тестирования базового подключения к Microsoft 365

## <a name="microsoft-365-architecture"></a>Архитектура Microsoft 365
<a name="BKMK_Architecture"> </a>

Microsoft 365 — это распределенное программное обеспечение, которое обеспечивает производительность и совместную работу с помощью разнообразных служб и приложений, таких как Exchange Online, SharePoint Online, Skype для бизнеса Online, Microsoft Teams, Exchange Online Protection, Office в браузере и многие другие. Хотя конкретные приложения Microsoft 365 могут иметь уникальные функции, которые применяются к клиентской сети и подключению к облаку, все они совместно используют некоторые основные субъекты, цели и шаблоны архитектуры. Эти принципы и шаблоны архитектуры для подключения являются типичными для многих других облаков SaaS и в то же время отличаются от типичных моделей развертывания для облаков платформы, а не служб и инфраструктуры, таких как Microsoft Azure.
  
Одной из важнейших архитектур Microsoft 365 (которые часто пропускают или ошибочно обрабатываются архитекторами сети) является то, что это действительно Глобальная распределенная служба в контексте того, как пользователи подключаются к ней. Расположение целевого клиента Microsoft 365 важно для изучения того, где данные клиента хранятся в облаке, но взаимодействие с корпорацией 365 Майкрософт не требует прямого подключения к дискам, содержащим данные. Взаимодействие пользователей с Microsoft 365 (включая производительность, надежность и другие важные характеристики качества) включает связь через высокораспределенные дверцы служб, которые размещаются в сотнях расположений Майкрософт по всему миру. В большинстве случаев лучшее взаимодействие с пользователями достигается благодаря тому, что клиентская сеть направляет запросы пользователей в ближайшую точку входа в службу Microsoft 365, а не подключается к Microsoft 365 с помощью точки выхода в центральном расположении или регионе.
  
Для большинства клиентов Microsoft 365 пользователей распространяется по многим расположениям. Для достижения наилучших результатов принципы, описанные в этом документе, следует Взгляните на точку с горизонтальным масштабированием (не горизонтальное масштабирование), в которой основное внимание уделяется оптимизации связи с ближайшим моментом присутствия в глобальной сети Майкрософт, а не в географическом расположении клиента Microsoft 365. По сути, это означает, что даже несмотря на то, что данные клиента Microsoft 365 могут храниться в определенном географическом расположении, взаимодействие с Microsoft 365 для этого клиента остается распределенным и может находиться в очень близком (сетевом) месте для каждого конечного пользователя, находящихся у клиента.
  
## <a name="microsoft-365-connectivity-principles"></a>Принципы подключения Microsoft 365
<a name="BKMK_Principles"> </a>

Корпорация Майкрософт рекомендует следующие принципы для достижения оптимального подключения и производительности Microsoft 365. Используйте эти принципы подключения Microsoft 365 для управления трафиком и получения наилучшей производительности при подключении к Microsoft 365.
  
Основная цель структуры сети — уменьшить задержку, уменьшив время кругового пути (RTT) из сети в глобальную сеть Майкрософт, магистраль сети общедоступной сети корпорации Майкрософт, которая соединяет все центры обработки данных Майкрософт с небольшой задержкой и точками входа облачного приложения, распространяющимися по всему миру. Вы можете узнать больше о глобальной сети Майкрософт, в которой [Корпорация Майкрософт создает свою быструю и надежную глобальную сеть](https://azure.microsoft.com/blog/how-microsoft-builds-its-fast-and-reliable-global-network/).
  
<a name="BKMK_P1"> </a>
### <a name="identify-and-differentiate-microsoft-365-traffic"></a>Определение и различение трафика Microsoft 365

![Определение трафика Microsoft 365](media/621aaec9-971d-4f19-907a-1ae2ef6d72fc.png)
  
Идентификация сетевого трафика Microsoft 365 это первый шаг, позволяющий отличать этот трафик от общего сетевого трафика, привязанного к Интернету. Подключение к Microsoft 365 может быть оптимизировано путем реализации различных подходов, таких как оптимизация маршрута сети, правила брандмауэра, параметры прокси-сервера браузера и обход устройств проверки сети для определенных конечных точек.
  
Предыдущее руководство по оптимизации Microsoft 365 разделяет конечные точки Майкрософт 365 на две категории: **обязательные** и **необязательные**. Так как конечные точки добавлены для поддержки новых служб и функций Microsoft 365, мы перестроили конечные точки Microsoft 365 на три категории: **optimize**, **allow**и **Default**. Рекомендации по каждой категории применяются ко всем конечным точкам в категории, что упрощает понимание и реализацию оптимизации.
  
Для получения дополнительных сведений о категориях конечных точек и методах оптимизации в Microsoft 365, обратитесь к разделу [Создание категорий конечных точек Office 365](office-365-network-connectivity-principles.md#BKMK_Categories) .
  
Теперь корпорация Майкрософт публикует все конечные точки Microsoft 365 в качестве веб-службы и предоставляет рекомендации по оптимальному использованию этих данных. Для получения дополнительных сведений о получении и работе с конечными точками Microsoft 365, ознакомьтесь со статьей [Office 365 URL-адреса и диапазоны IP-адресов](https://support.office.com/article/office-365-urls-and-ip-address-ranges-8548a211-3fe7-47cb-abb1-355ea5aa88a2?ui=en-US&amp;rs=en-US&amp;ad=US).
  
<a name="BKMK_P2"> </a>
### <a name="egress-network-connections-locally"></a>Локальная организация исходящего трафика для сетевых подключений.

![Локальная организация исходящего трафика для сетевых подключений.](media/b42a45be-1ab4-4073-a7dc-fbdfb4aedd24.png)
  
Локальная служба DNS и исходящий трафик из Интернета очень важны для снижения задержки подключения и обеспечения подключения пользователей к ближайшей точке входа в службы Microsoft 365 Services. В сложной сетевой топологии важно одновременно реализовать как локальные DNS, так и локальный выход в Интернет. Для получения дополнительных сведений о том, как Microsoft 365 маршрутизирует подключения клиентов к ближайшей точке входа, ознакомьтесь со статьей [подключение клиента](https://support.office.com/article/client-connectivity-4232abcf-4ae5-43aa-bfa1-9a078a99c78b).
  
Прежде чем приступать к облачным службам, таким как Microsoft 365, "подключение к Интернету для конечных пользователей в архитектуре сети" было относительно простым. Когда Интернет-службы и веб-сайты распространяются по всему миру, задержка между точками исходящего трафика Организации и любой заданной конечной точкой в основном является функцией географического расстояния.
  
В традиционной сетевой архитектуре все исходящие подключения к Интернету проходят через корпоративную сеть и отправляются из центрального расположения. Так как облачные решения корпорации Майкрософт устарели, для поддержки облачных служб с задержкой можно использовать распределенную сетевую архитектуру, доступную через Интернет. Глобальная сеть Майкрософт разработана для удовлетворения требований к задержке при использовании инфраструктуры "Передняя дверца распределенной службы", динамической структуры глобальных точек входа, которая направляет входящие подключения облачных служб в ближайшую точку входа. Это предназначено для уменьшения длины "Last миля" для клиентов Microsoft Cloud за счет эффективного сокращения маршрута между клиентом и облаком.
  
Корпоративные глобальные сети часто предназначены для получения сетевого трафика в Центральный офис компании для проверки перед выходом в Интернет, обычно через один или несколько прокси-серверов. На приведенной ниже схеме показана такая топология сети.
  
![Традиционная модель корпоративной сети](media/fc87b8fd-a191-47a7-9704-1e445599813a.png)
  
Так как Microsoft 365 работает в глобальной сети Майкрософт, которая включает серверы переднего плана по всему миру, сервер переднего плана, как правило, будет закрыт для размещения пользователя. Предоставляя локальные исходящий из Интернета и настраивая внутренние DNS-серверы для предоставления локального разрешения имен конечным точкам Microsoft 365, сетевой трафик, предназначенный для Microsoft 365, может подключаться к серверам переднего плана Microsoft 365 как можно ближе к пользователю. На схеме ниже показан пример топологии сети, которая позволяет пользователям, подключающимся из основного офиса, филиала и удаленных расположений, следовать кратчайшему маршруту до ближайшей точки входа Microsoft 365.
  
![Модель сети WAN с региональными точками выхода](media/4d4c07cc-a928-42b8-9a54-6c3741380a33.png)
  
Сокращение сетевого пути к точкам входа Microsoft 365 таким образом может повысить производительность подключения и взаимодействия с конечными пользователями в Microsoft 365, а также снизить влияние будущих изменений архитектуры сети на производительность и надежность Microsoft 365.
  
Кроме того, DNS-запросы могут привести к задержке, если отвечающий DNS-сервер является отдаленным или занятым. Можно минимизировать задержку разрешения имен путем подготовки локальных DNS-серверов в филиалах и убедитесь, что они настроены на кэширование DNS-записей соответствующим образом.
  
Несмотря на то, что региональные отработки могут хорошо работать в Microsoft 365, оптимальной моделью подключения будет всегда предоставляться сетевой выход в расположении пользователя, независимо от того, находится ли он в корпоративной сети или в удаленных расположениях, таких как домашние сети, Гостиницы, кафе и аэропорты. Эта Локальная модель прямого выхода представлена на приведенной ниже схеме.
  
![Локальная архитектура сети исходящего выхода](media/6bc636b0-1234-4ceb-a45a-aadd1044b39c.png)
  
Предприятия, которые применяют Microsoft 365, могут воспользоваться преимуществами архитектуры передней дверцы распределенной службы Microsoft Global Network, гарантируя, что подключения пользователей к Microsoft 365 будут иметь кратчайший возможный маршрут к ближайшей точке входа в глобальную сеть Майкрософт. Локальная архитектура сети исходящего трафика делает это, позволяя трафику Microsoft 365 маршрутизироваться по ближайшему выходу, независимо от местонахождения пользователя.
  
Архитектура локального выхода имеет следующие преимущества по сравнению с традиционной моделью:
  
- Обеспечивает оптимальную производительность Microsoft 365 за счет оптимизации длины маршрута. подключения конечных пользователей динамически направляются в ближайшую точку входа Microsoft 365 с помощью инфраструктуры "Передняя дверца" распределенной службы.
- Снижает нагрузку на корпоративную инфраструктуру сети, разрешая локальный выход.
- Обеспечивает безопасность подключений на обоих концах, используя функции безопасности конечной точки клиента и облачные функции безопасности.

<a name="BKMK_P3"> </a>
### <a name="avoid-network-hairpins"></a>Недопущение "узких" мест в сети.

![Избегайте развороты](media/ee53e8af-f57b-4292-a256-4f36733b263a.png)
  
Как правило, наиболее короткий и непосредственный маршрут между пользователями и ближайшими конечными точками Microsoft 365 обеспечивает наилучшую производительность. Сетевое перетасовки происходит, когда трафик WAN или VPN, привязанный к определенному назначению, сначала перенаправляется в другое промежуточное расположение (например, в стек безопасности, в брокере облачного доступа, на облачном веб-шлюзе), что позволяет использовать задержку и потенциальное перенаправление на территориально отдаленную конечную точку. Сетевые развороты также могут быть вызваны неэффективной маршрутизацией и неэффективной (удаленной) DNS-поиском.
  
Чтобы убедиться, что служба подключения к Microsoft 365 не подключается к сети развороты даже в локальном варианте выхода, убедитесь, что у поставщика услуг Интернета, который используется для выхода из Интернета для расположения пользователя, есть прямая связь с глобальной сетью Майкрософт в близком направлении к этому расположению. Кроме того, вы можете настроить маршрутизацию исходящих сообщений для отправки надежного трафика Microsoft 365, а не прокси-сервера или туннеля с помощью стороннего облачного или облачного поставщика безопасности, который обрабатывает трафик, связанный с Интернетом. Локальное разрешение имен DNS для конечных точек Microsoft 365 гарантирует, что в дополнение к прямой маршрутизации для пользовательских подключений используются ближайшие точки входа Майкрософт 365.
  
Если вы используете облачную сеть или службы безопасности для трафика Microsoft 365, убедитесь, что результат оценки перетасовки и его влияние на производительность Microsoft 365 понятны. Для этого можно проверить число и расположение расположений поставщиков услуг, по которым трафик пересылается по отношению к числу филиалов и глобальным точкам пиринга Майкрософт, а также по качеству отношения между сторонами подключения поставщика услуг к поставщику услуг Интернета и корпорацией Майкрософт, а также к снижению производительности в инфраструктуре поставщика услуг.
  
В связи с большим количеством распределенных расположений с точками входа Microsoft 365 и их близостью к конечным пользователям, маршрутизация трафика Microsoft 365 к любой сторонней сети или поставщику услуг безопасности может оказать неблагоприятное воздействие на подключения Microsoft 365, если сеть поставщика не настроена для оптимального пиринга Microsoft 365.
  
<a name="BKMK_P4"> </a>
### <a name="assess-bypassing-proxies-traffic-inspection-devices-and-duplicate-security-technologies"></a>Оценка обходных прокси-серверов, устройств проверки поступающих и дублирования технологий безопасности

![Обход прокси-серверов, устройств проверки поступающих и дублированных технологий безопасности](media/0131930d-c6cb-4ae1-bbff-fe4cf6939a23.png)
  
Корпоративным клиентам следует проанализировать методы обеспечения безопасности сети и риска, предназначенные специально для трафика Microsoft 365, и использовать функции безопасности Microsoft 365, чтобы снизить их зависимость от вмешательства, производительности и дорогостоящих технологий безопасности для сетевого трафика Microsoft 365.
  
Большинство корпоративных сетей применяют безопасность сети для трафика Интернета с помощью таких технологий, как прокси-серверы, проверка SSL, проверка пакетов и защита от потери данных. Эти технологии обеспечивают важное снижение рисков для универсальных Интернет – запросов, но могут значительно снизить производительность, масштабируемость и качество работы конечного пользователя при применении к конечным точкам Microsoft 365.
  
<a name="BKMK_WebSvc"> </a>
#### <a name="office-365-endpoints-web-service"></a>Веб-служба конечных точек Office 365

Администраторы Microsoft 365 могут использовать вызов сценариев или REST для использования структурированного списка конечных точек из веб-службы конечных точек Office 365 и обновления конфигураций брандмауэров периметра и других сетевых устройств. Это гарантирует, что трафик, связанный с Microsoft 365, будет идентифицирован, правильно обрабатываться и управляться иначе, чем сетевой трафик, привязанный к общим и часто неизвестным веб-сайтам Интернета. Дополнительные сведения о том, как использовать веб-службу конечных точек Office 365, можно найти в статье [URL-адреса и диапазоны IP-адресов office 365](https://support.office.com/article/office-365-urls-and-ip-address-ranges-8548a211-3fe7-47cb-abb1-355ea5aa88a2?ui=en-US&amp;rs=en-US&amp;ad=US).
  
#### <a name="pac-proxy-automatic-configuration-scripts"></a>Сценарии автоматической настройки прокси-сервера (PAC)
<a name="BKMK_WebSvc"> </a>

Администраторы Microsoft 365 могут создавать сценарии с автоматической настройкой прокси-сервера (PAC), которые можно доставлять на компьютеры пользователей с помощью WPAD или объекта групповой политики. Сценарии PAC могут использоваться для обхода прокси-серверов для запросов Microsoft 365 от пользователей WAN или VPN, что позволяет трафику Microsoft 365 использовать прямые подключения к Интернету, а не проходить в корпоративную сеть.
  
#### <a name="microsoft-365-security-features"></a>Функции безопасности Microsoft 365
<a name="BKMK_WebSvc"> </a>

Майкрософт является прозрачной для безопасности центра данных, эксплуатационной безопасности и снижения риска на серверах Microsoft 365 и конечных точках сети, которые они представляют. Встроенные функции безопасности Microsoft 365 доступны для снижения риска нарушения безопасности сети, например защиты от потери данных, защиты от вирусов, многофакторной проверки подлинности, блокировки клиента, расширенной защиты от угроз, Microsoft 365 Threat Intelligence, Microsoft 365 надежный показатель, Exchange Online Protection и безопасность сети DDOS.
  
Дополнительные сведения о центре обработки данных Майкрософт и глобальной сети безопасности можно найти в [центре управления безопасностью Майкрософт](https://www.microsoft.com/trustcenter/security).
  
## <a name="new-office-365-endpoint-categories"></a>Новые категории конечных точек Office 365
<a name="BKMK_Categories"> </a>

Конечные точки Office 365 представляют собой разнообразный набор сетевых адресов и подсетей. Конечными точками могут быть URL-адреса, IP-адреса или диапазоны IP-адресов, а некоторые конечные точки указаны с определенными портами TCP/UDP. URL-адреса могут быть либо полными доменными именами, например *Account.Office.NET*, либо подстановочным URL-адресом, например * \* . Office365.com*.
  
> [!NOTE]
> Расположения конечных точек Office 365 в сети напрямую не связаны с расположением данных клиента Microsoft 365. По этой причине клиенты должны взглянуть на Microsoft 365 в качестве распределенной и глобальной службы и не должны пытаться блокировать сетевые подключения к конечным точкам Office 365 на основе географического критерия.
  
В предыдущих руководствах по управлению трафиком Microsoft 365 конечные точки были упорядочены по двум категориям: **обязательным** и **необязательным**. Для конечных точек в каждой категории требуется разная оптимизация в зависимости от важности службы, и многие клиенты столкнулись с трудностями при выключе приложения одной и той же оптимизации сети к полному списку URL-адресов и IP-адресов Office 365.
  
В новой модели конечные точки разделяются на три категории, **оптимизируются**, **разрешаются**и **по умолчанию**, предоставляя основанную на приоритетах сводную таблицу, в которой разделяются действия по оптимизации сети, чтобы реализовать лучшие улучшения производительности и рентабельность инвестиций. Конечные точки консолидируются в приведенных выше категориях в соответствии с чувствительностью эффективного взаимодействия с пользователем к качеству сети, тому, а также конверту и производительности, а также простоте реализации. Рекомендуемые способы оптимизации могут быть применены ко всем конечным точкам в определенной категории.
  
- **Оптимизация** конечных точек необходима для подключения к каждой службе Office 365 и предоставления более 75% пропускной способности Office 365, подключений и объема данных. Эти конечные точки представляют сценарии Office 365, наиболее чувствительные к производительности сети, задержкам и доступности. Все конечные точки размещаются в центрах обработки данных Майкрософт. Частота изменений конечных точек в этой категории должна быть меньше, чем для конечных точек в других двух категориях. Эта категория включает небольшое (в порядке, в 10) набор URL-адресов ключей и определенный набор IP-подсетей, выделенных для основных рабочих нагрузок Office 365, таких как Exchange Online, SharePoint Online, Skype для бизнеса Online и Microsoft Teams.

    Сокращенный список хорошо заданных критически важных конечных точек поможет вам быстрее и легче спланировать и реализовать высококачественную оптимизацию сети для этих целей.

    Примеры *оптимизации* конечных точек: *https://outlook.office365.com* , *https:// \<tenant\> . SharePoint.com*и *https://. \<tenant\> My.SharePoint.com*.

    Методы оптимизации включают:

  - Обход конечных точек *оптимизации* на сетевых устройствах и службах, выполняющих перехват трафика, расшифровку SSL, глубокую проверку пакетов и фильтрацию содержимого.
  - Обход локальных прокси-устройств и облачных прокси-служб обычно используется для общего просмотра в Интернете.
  - Определите приоритеты для оценки этих конечных точек как полностью доверенные для сетевой инфраструктуры и пограничной системы.
  - Назначение приоритетов уменьшению или удалению глобальной сети WAN и упрощение прямого распределенного выхода из Интернета для этих конечных точек как можно ближе к пользователям и расположениям филиалов.
  - Упрощение прямого подключения к этим конечным точкам для VPN-пользователей путем реализации раздельного туннелирования.
  - Убедитесь, что IP-адреса, возвращаемые разрешением имен DNS, совпадают с путями выхода маршрутизации для этих конечных точек.
  - Определите приоритеты для этих конечных точек для интеграции SD/WAN для прямой, минимальной задержки маршрутизации в ближайшую точку подключения к Интернету для глобальной сети Майкрософт.

- **Разрешить** конечные точки необходимы для подключения к определенным функциям и службам Office 365, но они не чувствительны к производительности сети и задержкам, как и в категории *Оптимизация* . Общий объем сети для этих конечных точек с точки зрения пропускной способности и числа подключений также уменьшается. Эти конечные точки выделены для Office 365 и размещаются в центрах обработки данных Майкрософт. Они представляют собой обширный набор Microsoft Office 365 Micro-Services и их зависимостей (в порядке с URL-адресами ~ 100), которые должны меняться на более высоких частотах, чем в категории *optimize* . Не все конечные точки в этой категории связаны с определенными выделенными IP-подсетями.

    Оптимизация сети для *разрешения* конечных точек может повысить удобство работы пользователей Office 365, но некоторые пользователи могут выбрать более узкое количество таких оптимизаций, чтобы минимизировать изменения в сети.

    К примерам *разрешенных* конечных точек относятся *https:// \* . Protection.Outlook.com* и *https://accounts.accesscontrol.windows.net* .

    Методы оптимизации включают:

  - Обход *разрешенных* конечных точек на сетевых устройствах и службах, выполняющих перехват трафика, расшифровку SSL, глубокую проверку пакетов и фильтрацию содержимого.
  - Определите приоритеты для оценки этих конечных точек как полностью доверенные для сетевой инфраструктуры и пограничной системы.
  - Назначение приоритетов уменьшению или удалению глобальной сети WAN и упрощение прямого распределенного выхода из Интернета для этих конечных точек как можно ближе к пользователям и расположениям филиалов.
  - Убедитесь, что IP-адреса, возвращаемые разрешением имен DNS, совпадают с путями выхода маршрутизации для этих конечных точек.
  - Определите приоритеты для этих конечных точек для интеграции SD/WAN для прямой, минимальной задержки маршрутизации в ближайшую точку подключения к Интернету для глобальной сети Майкрософт.

- Конечные точки **по умолчанию** представляют службы и зависимости Office 365, для которых не требуется оптимизация, и могут обрабатываться сетями клиентов как обычный трафик, связанный с Интернетом. Некоторые конечные точки в этой категории не могут размещаться в центрах обработки данных Майкрософт. Примеры включают *https://odc.officeapps.live.com* и *https://appexsin.stb.s-msn.com* .

Для получения дополнительных сведений о методах оптимизации сети Office 365, ознакомьтесь со статьей [Управление конечными точками office 365](managing-office-365-endpoints.md).
  
## <a name="comparing-network-perimeter-security-with-endpoint-security"></a>Сравнение безопасности периметра сети с безопасностью конечной точки
<a name="BKMK_SecurityComparison"> </a>

Для традиционной безопасности сети необходимо защитить периметр сети Организации от вторжения и проникновения злоумышленников. По мере того как организации принимают Microsoft 365, некоторые сетевые службы и данные частично или полностью переносятся в облако. Как и для любых фундаментальных изменений в архитектуре сети, этот процесс требует повторной оценки безопасности сети, которая учитывает новые факторы:
  
- По мере того как облачные службы приняты, сетевые службы и данные распределяются между локальными центрами обработки данных и облаком, а безопасность периметра более недостаточна.
- Удаленные пользователи подключаются к корпоративным ресурсам как в локальных центрах обработки данных, так и в облаке из неуправляемых расположений, таких как домашние и Гостиницы, а также кафе.
- Встроенные функции безопасности все чаще встраиваются в облачные службы и потенциально могут дополнять или заменять существующие системы безопасности.

Корпорация Майкрософт предлагает широкий спектр функций безопасности Microsoft 365 и предоставляет рекомендации по обеспечению безопасности, которые могут помочь обеспечить безопасность данных и сети для Microsoft 365. Ниже приведены рекомендуемые рекомендации.
  
- **Использование многофакторной проверки подлинности (MFA)** MFA добавляет дополнительный уровень защиты в стратегию надежных паролей, требуя от пользователей подтвердить телефонный звонок, текстовое сообщение или уведомление приложения на смарт-телефоне после правильного ввода пароля.

- **Использование Microsoft Cloud App Security** Настройка политик для отслеживания аномальных действий и выполнения действий над ним. Настройте оповещения с помощью Microsoft Cloud App Security, чтобы администраторы могли просматривать необычные или рискованные действия пользователей, такие как загрузка больших объемов данных, несколько неудачных попыток входа или подключения из неизвестных или небезопасных IP-адресов.

- **Настройка защиты от потери данных (DLP)** DLP позволяет определять конфиденциальные данные и создавать политики, которые помогают предотвратить случайное или намеренное совместное использование данных пользователями. DLP работает в Microsoft 365, включая Exchange Online, SharePoint Online и OneDrive, чтобы пользователи могли оставаться соответствующими, не прерывая рабочий процесс.

- **Использование защищенного хранилища клиентов** Как администратор Microsoft 365, вы можете использовать защищенное хранилище клиента для управления доступом к данным специалистом службы поддержки Майкрософт во время сеанса справки. В случаях, если сотрудник службы поддержки обратится к вашим данным, чтобы устранить неполадки, защищенное хранилище позволит вам одобрить или отклонить его запрос на доступ.

- **Использование оценки безопасности Office 365** Средство аналитики безопасности, которое позволяет повысить риск снижения риска. Показатель безопасности рассматривает параметры и действия Microsoft 365 и сравнивает их с базовым планом, установленным корпорацией Майкрософт. Вы получите оценку на основе соответствия рекомендациям по обеспечению безопасности.

Комплексный подход к усиленной безопасности должен учитывать следующее:
  
- Изменяйте выделение с защитой от периметра к безопасности конечной точки, применяя функции безопасности на основе облака и клиента Office.
  - Уменьшение периметра безопасности для центра обработки данных
  - Включение доверительного отношения доверия для пользовательских устройств в Office или в удаленных расположениях
  - Сосредоточьтесь на защите расположения данных и расположения пользователя
  - Управляемые пользовательские компьютеры имеют более высокий уровень доверия с безопасностью конечной точки
- Комплексное управление информационной безопасностью (не только на периметре)
  - Переопределение параметров глобальной сети и глобальной сети периметра, позволяющих надежному трафику обходить устройства безопасности и разделять неуправляемые устройства гостевым сетям Wi/Fi
  - Снижение требований к безопасности сети для корпоративной глобальной сети
  - Некоторые устройства безопасности сети периметра сети, такие как брандмауэры, по-прежнему являются обязательными, но нагрузка уменьшается
  - Обеспечивает возможность локального выхода для трафика Microsoft 365
- Улучшения можно устранить постепенно, как описано в разделе " [добавочная оптимизация](office-365-network-connectivity-principles.md#BKMK_IncOpt) ". Некоторые способы оптимизации могут обеспечить лучшую рентабельность и выгод в зависимости от архитектуры сети, а также выбрать оптимизацию, которая будет наиболее осмысленной для вашей организации.

Для получения дополнительных сведений о безопасности и совместимости Microsoft 365 ознакомьтесь со статьей [microsoft 365 Security](https://docs.microsoft.com/microsoft-365/security) and [Microsoft 365 Security](https://docs.microsoft.com/microsoft-365/compliance).
  
## <a name="incremental-optimization"></a>Добавочная оптимизация
<a name="BKMK_IncOpt"> </a>

Мы выстроили идеальную модель сетевого подключения для SaaS, описанную выше в этой статье, но во многих крупных организациях с историческими сетевыми архитектурами не будет целесообразно напрямую вносить все эти изменения. В этом разделе мы обсудим ряд добавочных изменений, которые помогут повысить производительность и надежность Microsoft 365.
  
Методы, которые будут использоваться для оптимизации трафика Microsoft 365, зависят от топологии сети и реализованных сетевых устройств. Для больших предприятий с множеством расположений и сложными методиками обеспечения безопасности сети необходимо разработать стратегию, включающую большинство или все принципы, перечисленные в разделе [принципы подключения Microsoft 365](office-365-network-connectivity-principles.md#BKMK_Principles) , в то время как небольшие организации могут использовать только один или два.
