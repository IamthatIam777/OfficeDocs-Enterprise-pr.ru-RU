# <a name="build-from-the-ground-up"></a>Создание с чистого листа

 **Сводка:** Получите подробные сведения об набор облачных хранилища стандартные блоки, которые можно использовать для создания службы хранилища или решения.
  
Что можно или нужно в случае, если вы создаете решения для хранения с чистого листа:
  
- Позволяют создавать свои собственные решения хранения с нуля. 
    
- Требуется программирование с использованием API-интерфейсы REST.
    
- Предоставьте максимум в настройке и гибкости.
    
В следующих разделах подробно описан каждый из вариантов решений для хранения, созданных с чистого листа.
  
## <a name="azure-storage-files"></a>Служба хранилища Azure (файлы)

### <a name="features"></a>Возможности

- Более простое перемещение приложений прежних версий в облако.
    
- Хранилище BLOB-объектов, предпочтительное для новых приложений.
    
- Возможность подключения из виртуальной машины Azure.
    
- Возможность подключения локальной среды с помощью SMB 3.0.
    
- Работает с ОС Linux и Windows.
    
- Не поддерживает проверку подлинности на основе AD Azure или списки управления доступом (ключи хранилища Azure учетной записи проверки подлинности и авторизованные доступ к общей папке)
    
### <a name="common-uses"></a>Стандартные варианты использования

- Перенос приложений прежних версий, зависящих от файловых ресурсов, в облако.
    
- Совместное использование средств разработки и тестирования.
    
- Распределенные приложения могут сохранять журналы, диагностические данные и аварийные дампы.
    
### <a name="key-storage-scenarios"></a>Основные сценарии использования хранилища

- Резервное копирование файлов
    
### <a name="resources"></a>Ресурсы

Дополнительные сведения вы найдете [здесь](https://msdn.microsoft.com/library/azure/dn166972.aspx).
  
Сведения о ценах см. [здесь](http://azure.microsoft.com/pricing/details/storage/).
  
## <a name="azure-storage-blobs"></a>Служба хранилища Azure (большие двоичные объекты)

### <a name="features"></a>Возможности

- Каждой учетной записи хранения может содержать до 500 ТБ (одна подписка может иметь несколько учетных записей хранилища)
    
- Учетные записи хранения упорядочены в контейнеры, которые можно защитить и которые могут содержать большие двоичные объекты.
    
- Блочные BLOB-объекты оптимизированы для потоковой передачи и хранения облачных объектов размером до 200 ГБ.
    
- Страничные BLOB-объекты оптимизированы для представления дисков PaaS и операций произвольной записи и могут иметь размер до 1 ТБ.
    
- Большие двоичные объекты добавления оптимизированы для операций добавления и могут иметь размер до 195 ГБ.
    
- Хранилище класса Premium имеет более высокие показатели количества операций ввода-вывода в секунду из-за использования SSD.
    
### <a name="common-uses"></a>Стандартные варианты использования

- Резервные копии файлов, компьютеров, баз данных и устройств изображения и текст для веб-приложений
    
- Данные конфигурации для облачных приложений.
    
- Большие данные, например журналы и другие большие наборы данных.
    
- Azure использует хранилище BLOB-объектов для собственных служб, например HDInsight и дисков виртуальных машин.
    
### <a name="key-storage-scenarios"></a>Основные сценарии использования хранилища

- Управление данными.
    
### <a name="resources"></a>Ресурсы

Дополнительные сведения вы найдете [здесь](https://msdn.microsoft.com/library/azure/dd179376.aspx).
  
Сведения о ценах см. [здесь](http://azure.microsoft.com/pricing/details/storage/).
  
## <a name="azure-storage-queues"></a>Служба хранилища Azure (очереди)

### <a name="features"></a>Возможности

- Учетная запись хранения может содержать любое количество очередей.
    
- Очередь может содержать любое количество сообщений (пока учетная запись хранения не будет заполнена).
    
- Если сообщения в очереди не извлечены или не удалены приложением, они будут автоматически удалены по истечении семи дней.
    
- Сообщения могут иметь размер до 64 КБ.
    
- Защита на уровне учетной записи хранения.
    
- Очереди предназначены для передачи управляющих сообщений, а не необработанных данных.
    
### <a name="common-uses"></a>Стандартные варианты использования

- Подготовка невыполненной работы для асинхронной обработки.
    
- Обработка сообщений в журналах.
    
- Разъединение приложений.
    
### <a name="key-storage-scenarios"></a>Основные сценарии использования хранилища

- Распределение событий.
    
### <a name="resources"></a>Ресурсы

Дополнительные сведения вы найдете [здесь](https://msdn.microsoft.com/library/azure/dd179353.aspx).
  
Сведения о ценах см. [здесь](http://azure.microsoft.com/pricing/details/storage/).
  
## <a name="azure-storage-tables"></a>Служба хранилища Azure (таблицы)

### <a name="features"></a>Возможности

- Решение лучше всего подходит для частично структурированных наборов данных.
    
- Обычно требует более низких затрат по сравнению с традиционными SQL.
    
- Очень быстрый интерфейс, если запрос для ключа медленно, если запрос для значения
    
- Широкие возможности масштабирования; любое количество таблиц вплоть до максимального объема дискового пространства учетной записи хранения.
    
- Доступ через REST API, ограниченный протокол oData, .NET.
    
- Значения должны быть сериализованы.
    
### <a name="common-uses"></a>Стандартные варианты использования

- Данные пользователей для веб-приложений.
    
- Адресные книги.
    
- Сведения об устройствах.
    
### <a name="key-storage-scenarios"></a>Основные сценарии использования хранилища

- Управление данными.

## <a name="microsoft-azure-storage-recommendations"></a>Рекомендации по службе хранилища Microsoft Azure

Когда вы создаете свое специальное решение для хранения с использованием службы хранилища Azure, помните об этом:
  
- Для увеличения масштабируемости, дискового пространства (более 100 ТБ) или пропускной способности (более 5000 операций в секунду) используйте несколько учетных записей хранения.
    
- Запланируйте возможность добавления дополнительных учетных записей хранения путем изменения конфигурации, а не кода.
    
- Тщательно выберите функции секционирования для хранилища таблиц, чтобы достичь необходимого уровня производительности при вставке данных и обработке запросов.
    
- Выбирайте короткие имена столбцов для свойств таблиц, так как метаданные (имена свойств) хранятся в самих таблицах (имена столбцов включаются в данные строк, размер которых не должен превышать 1 МБ).
    
- По возможности используйте пакетные операции в хранилище.
    
- Выполняйте агрессивное кэширование информации, содержащейся в базе данных конфигурации, добавляя ее в распределенный кэш.
    
- Если производительность или надежность приложения зависит от доступности определенного сегмента данных в кэше, ваше приложение должно отклонять входящие запросы до тех пор, пока кэш не будет повторно заполнен.
    
- Сегментируйте данные по вертикали (по таблицам) либо по горизонтали (разбивайте таблицы на несколько сегментов), чтобы распределить нагрузку в нескольких базах данных.
